## Blocking Gap

### GAP-1 — Controller dashboard UI (inspection.html) missing from src/main/resources/templates

**Severity: BLOCKING (CTRL-01, CTRL-02, CTRL-03, CTRL-04)**

`inspection.html` exists only under `target/classes/templates/` — it is absent from
`src/main/resources/templates/`. The file present in `target/` is a compiled artifact
from a previous build and will be overwritten or lost on `mvn clean`.

Additionally, `ViewController.java` has **no `@GetMapping("/inspection")` route**.
The navigation links in `index.html` (line 465) and `inspection-map.html` (line 67)
both href to `/inspection`, but the server will return a Thymeleaf 404 (no view named
"inspection") because `ViewController` does not map that path.

**Break point:** browser navigates to `/inspection` → Spring MVC cannot resolve view
`inspection` → Whitelabel Error Page (404). All four CTRL-* E2E paths fail at the first
step (loading the dashboard).

**Fix required:**
1. Copy `target/classes/templates/inspection.html` to `src/main/resources/templates/inspection.html`
2. Add `@GetMapping("/inspection") public String inspection() { return "inspection"; }` to `ViewController.java`

## Moderate Gaps

### GAP-2 — `controller.signup.code` not set in Docker Compose or application.properties

**Severity: MODERATE (AUTH-02)**

`AuthService` reads `${controller.signup.code:#{null}}`. When the property is null or
absent, any registration attempt with a `signupCode` field raises
`"Invalid registration request"` (correct fail-safe), but no CONTROLLER account can
be self-registered at runtime.

`application.properties` contains no `controller.signup.code` entry. `docker-compose.yml`
injects no `CONTROLLER_SIGNUP_CODE` env var into the `restaurant-app` container.

The two seed accounts (`controller_test / Test1234!`) created by `DataSeeder` fill the
gap for testing, but the documented AUTH-02 user flow (controller registers via shared
code) is non-functional without a manual env var injection outside version control.

**Fix:** Add `CONTROLLER_SIGNUP_CODE=<value>` to `docker-compose.yml` under the app
service's `environment:` block (or document the manual step explicitly).

### GAP-3 — `/hygiene-radar` view route missing from ViewController

**Severity: MODERATE (navigation, no direct REQ mapping)**

`index.html` line 463 has a persistent nav link `<a href="/hygiene-radar">`.
`ViewController` has no `@GetMapping("/hygiene-radar")`. Clicking the link returns
a 404 / Whitelabel error. The API endpoint `GET /api/restaurants/hygiene-radar`
is correctly implemented, but the Thymeleaf template `hygiene-radar.html` (present
in `target/classes/templates/`) is not reachable from the controller.

**Fix:** Add `@GetMapping("/hygiene-radar") public String hygieneRadar() { return "hygiene-radar"; }` to `ViewController` and move `hygiene-radar.html` to `src/main/resources/templates/`.

## Notes (Non-Blocking)

### NOTE-1 — Restaurant detail page fetches without auth token

`restaurant.html` issues its primary fetch (`GET /api/restaurants/{restaurantId}`)
without the `Authorization` header (line 425-459: plain `fetch()`, not `fetchWithAuth()`).
This is intentional — `/api/restaurants/**` is public per `SecurityConfig` — and works
correctly for anonymous browsing (CUST-02). However the bookmark section uses
`fetchWithAuth()` only when `accessToken` is present in localStorage, which is correct.
No action required; behavior is as designed.

### NOTE-2 — inspection-map.html is accessible to anonymous users

`ViewController.inspectionMap()` maps `/inspection-map` with no auth guard.
`SecurityConfig` has `.anyRequest().permitAll()` for non-API routes. An anonymous
user can visit `/inspection-map` and the map loads via the public
`GET /api/restaurants/map-points` endpoint. This is consistent with CUST-03
("customer browses grade-colored interactive map") and the stated Phase 3 scope
comment in `SecurityConfig`. No issue.

### NOTE-3 — `my-bookmarks.html` has client-side auth redirect only

`/my-bookmarks` has no server-side auth guard (covered by `anyRequest().permitAll()`).
The client-side guard at `loadBookmarks()` line 206 redirects to `/login` if no
`accessToken` is in localStorage. An unauthenticated direct GET to
`/my-bookmarks` renders the spinner briefly then redirects. The API call
`GET /api/users/me/bookmarks` is correctly protected at the server level
(requires authenticated). Acceptable for SPA-style flow.

### NOTE-4 — login.html register form uses `prompt()` dialogs

The CUSTOMER registration UI (login.html lines 135-165) uses browser `prompt()` calls
rather than a form. There is no `signupCode` prompt, so customers correctly receive
`ROLE_CUSTOMER`. Controller registration via `signupCode` has no UI path exposed to
regular users. The missing signup code (GAP-2) means controller self-registration via
UI is currently impossible regardless of the UI approach.
