<objective>
Eliminate all hardcoded secrets from application source and configuration, add a fail-fast startup assertion on JWT_SECRET, and create the test infrastructure (Wave 0 gaps) that guards the rest of Phase 13 execution.

Purpose: The application currently starts silently with an insecure hardcoded JWT key. This plan makes that impossible — startup fails with a descriptive message if JWT_SECRET is absent or too short. It also creates the test fixtures needed so the assertion does not break existing JwtUtilTest tests.

```java
// Priority order in getProperty(): System.getenv(envKey) → dotenv.get(envKey) → properties.getProperty(key, defaultValue)
// envKey = key.replace(".", "_").toUpperCase()  →  "jwt.secret" maps to "JWT_SECRET"

public static String getJwtSecret() {
    return getProperty("jwt.secret", "changeit-please-change-it");
    // MUST CHANGE: replace default with IllegalStateException assertion (D-02, D-03)
}

private static String getProperty(String key, String defaultValue) {
    String envKey = key.replace(".", "_").toUpperCase();
    String envValue = System.getenv(envKey);
    if (envValue != null) return envValue;
    if (dotenv != null) {
        String dotenvValue = dotenv.get(envKey, null);
        if (dotenvValue != null) return dotenvValue;
    }
    return properties.getProperty(key, defaultValue);
}

// IMPORTANT: System.setProperty() writes to system properties NOT environment variables.
// AppConfig.getProperty() checks System.getenv() only — System.setProperty() does NOT bypass the assertion.
// The only reliable way to inject "jwt.secret" in unit tests is to set it directly in the
// AppConfig.properties static field via reflection (established project pattern — see STATE.md).
private static final Properties properties = new Properties();  // accessible via reflection
```

```java
private String savedJwtSecret;

@After
public void restoreJwtSecret() throws Exception {
    Properties props = getAppConfigProperties();
    if (savedJwtSecret == null) {
        props.remove("jwt.secret");
    } else {
        props.setProperty("jwt.secret", savedJwtSecret);
    }
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| env → AppConfig | JWT_SECRET and signup codes cross the OS/container boundary into the JVM via System.getenv() and dotenv |
| .env file → developer machine | Secrets in .env are visible to any process with filesystem read access |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01 | Information Disclosure | `application.properties` — jwt.secret hardcoded value | mitigate | Remove `jwt.secret=dev-only-insecure-key-...` line entirely; replace with comment pointing to JWT_SECRET env var |
| T-13-02 | Elevation of Privilege | `AppConfig.getJwtSecret()` — default fallback `"changeit-please-change-it"` allowed startup with a known weak key | mitigate | Replace default with `IllegalStateException` — application refuses to start if JWT_SECRET absent or < 32 chars |
| T-13-03 | Information Disclosure | `.env.example` committed to repository | accept | .env.example contains no real secrets (only `<FILL_IN>` placeholders); it is the documentation artifact and must be committed |
| T-13-04 | Information Disclosure | `.env` file with real secrets on developer filesystem | mitigate | Ensure `.gitignore` includes `.env` entry; `.env.example` header explicitly warns not to commit `.env` |
| T-13-05 | Elevation of Privilege | JWT signing key length < 256 bits (32 bytes) accepted silently | mitigate | Assertion enforces `secret.length() >= 32` — catches short keys before JwtUtil is constructed |
</threat_model>
